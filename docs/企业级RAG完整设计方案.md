# 真实企业级 RAG 完整落地实施清单（制造业集团内部知识库项目）

## 项目背景（真实线下制造集团场景，可直接复用面试 / 实战）

### 企业基础信息

某大型装备制造集团，员工 8000\+，下属 5 个生产基地、研发院、售后维保部、采购部、安全合规部； 现有痛点：

1. 内部资料分散：研发图纸说明、设备维修手册、安全操作规程、零部件参数、采购标准、历史故障工单、质检规范散落在共享盘、飞书文档、本地电脑、纸质扫描合同；

2. 新人培训成本极高，售后工程师外出抢修查手册要翻几百份 PDF，平均单次故障排查耗时 40 分钟；

3. 传统关键词搜索经常搜不到专业工艺术语，纯大模型问答凭空编造设备参数，出现过误导维修导致停机事故；

4. 合规要求：所有内部技术资料禁止出内网，不能使用 SaaS 在线 AI 工具，必须私有化部署，文档按部门设置访问权限，所有问答留审计日志满足安监审查；

5. 业务诉求：搭建统一内部 AI 知识库，实现「员工问答机器人 \+ 售后工单自动分析 \+ 工艺文档智能检索」三合一 RAG 系统。

### 最终技术选型（企业真实落地组合）

Dify（私有化 LLMOps 平台）\+ Milvus 分布式向量库 \+ PGVector 辅助存储 \+ BGE\-M3 私有化 Embedding \+ 本地部署通义千问 72B 企业版 \+ RAGFlow 做扫描件 / OCR 解析 配套：Docker Compose 单机部署（初期 POC），后期迁移 K8s 集群；对接企业统一登录 SSO、飞书内部 IM、工单 MySQL 业务库。

# 全流程落地实施步骤（分 6 大阶段，每个阶段含真实工作内容、交付物、工程师工作）

## 阶段 1：需求调研 \& 范围确认（1\~3 天，产品 \+ AI 开发 \+ 业务部门负责人协同）

### 1\.1 业务访谈（5 大核心使用部门）

1. 售后维保部 需求：输入设备故障现象，自动匹配维修步骤、对应备件型号、历史同类故障解决方案，支持导出维修方案；禁止输出不存在的零部件参数；

2. 研发部 需求：检索零部件设计规范、材料耐受参数、图纸配套文字说明，支持多文档对比；研发资料仅研发人员可见；

3. 安全生产部 需求：快速调取设备安全操作规范、事故处置流程、安监检查标准，回答必须标注文档来源页码；

4. 生产车间 需求：操作工快速查询流水线操作步骤、质检判定标准，输出简洁短句，适配车间大屏；

5. 采购部 需求：检索供应商准入标准、零部件采购技术要求、历史招标文档。

### 1\.2 约束条件梳理（企业硬性规则，开发必须落地）

1. 部署约束：全内网离线环境，禁止访问公网；

2. 权限约束：RBAC 多租户隔离，研发文档售后不可见，涉密资料仅管理员开放；

3. 数据约束：PDF、扫描件、Word、Excel、飞书在线文档、历史工单文本全部纳入；扫描图纸多栏、手写批注需完整识别；

4. 合规约束：每一条问答记录存储 6 个月，包含提问人、提问时间、检索文档、AI 回答，支持安监导出审计日志；

5. 性能约束：单次问答全链路响应≤3s，支持同时在线 200 人使用。

### 1\.3 输出交付物

《RAG 项目需求规格说明书》《知识库文档分类目录》《权限分级清单》

### AI 开发工程师工作

记录业务检索高频术语、识别文档难点（大量扫描件、多栏工艺手册），初步确定向量库、文档解析工具选型。

## 阶段 2：POC 验证（3\~7 天，AI 开发独立完成，验证方案可行性）

### 2\.1 POC 数据准备

抽取各部门典型测试文档：维修手册 10 份、扫描安全规范 5 份、零部件参数 Excel、历史故障工单 200 条。

### 2\.2 搭建最小可用 RAG 链路

1. 本地 Docker 部署 Dify \+ Milvus 单机实例；

2. 接入私有化 Embedding 模型 BGE\-M3、本地大模型；

3. 接入 RAGFlow 处理扫描 PDF，输出干净文本给 Dify；

4. 配置基础 RAG 策略：语义分块、BM25 \+ 向量混合召回、BGE\-Rerank 重排；

### 2\.3 核心验证指标（企业验收标准）

1. 专业故障问题召回准确率≥92%；

2. 无文档场景 AI 拒绝编造答案，幻觉率趋近 0；

3. 扫描件文字识别完整，表格参数可精准检索；

4. 响应速度满足≤3s 要求。

### 2\.4 POC 输出交付物

《POC 测试报告》《技术选型最终方案》《硬件资源评估清单》

### 落地真实踩坑（企业现场遇到）

直接用 Dify 原生解析扫描图纸识别乱码、表格丢失；解决方案：前置 RAGFlow 做文档预处理流水线。

## 阶段 3：数据 ETL 流水线搭建（7\~15 天，AI 开发 \+ 后端协同）

本阶段是企业 RAG 工作量最大环节，解决企业杂乱异构数据问题。

### 3\.1 数据源对接开发

1. 静态文件源：企业共享盘挂载目录，开发定时增量同步脚本；新增文档自动入库，重复文档去重；

2. 在线文档源：对接飞书开放 API，自动拉取部门知识库文档；

3. 业务数据库源：开发定时任务读取 MySQL 售后故障工单，转为文本存入知识库；

4. 扫描文件分流：PDF 自动判断是否为图片扫描件，图片类转发 RAGFlow 做 OCR、表格结构化。

### 3\.2 文档预处理标准化流程（固定流水线）

原始文件 → 格式解析 → 水印 / 页眉页脚清洗 → 表格提取为文本块 → 语义分割 Chunk → 绑定元数据（部门、文档类型、日期、权限标签）→ 向量化存入 Milvus 分块策略（制造业专用）：

- 普通文字文档：500 字符 Chunk，重叠 100 字符；

- 工艺 / 维修手册：父子分块，父块 3000 字符用于检索，子块 500 字符送入 LLM 生成答案；

- 零部件参数表格：单行为独立 Chunk，避免参数混杂丢失。

### 3\.3 元数据体系设计（权限过滤核心）

每条文本块携带标签：`dept(研发/售后/安全)`、`doc_type(手册/工单/规范)`、`secret_level(普通/涉密)`，检索时根据登录用户身份自动过滤无权限文档。

### 交付物

《数据同步接口文档》《文档预处理脚本包》《知识库元数据规范》

## 阶段 4：Dify 平台深度定制开发（10\~20 天，核心开发阶段）

### 4\.1 基础平台部署与优化

1. 生产环境 Docker Compose 高可用部署，调整连接池、向量检索并发参数；

2. 对接企业 SSO 统一登录，替换 Dify 原生账号系统；

3. 配置多知识库隔离：研发库、售后库、安全库、采购库，独立权限开关；

### 4\.2 RAG 核心策略配置（生产级调优）

1. 多路混合召回：Milvus 向量检索 \+ BM25 稀疏检索，结果合并去重；

2. 重排配置：召回 Top20，重排后仅保留 Top6 高质量片段送入 LLM；

3. Query 优化工作流：用户提问先经大模型改写、拆分多子问题并行检索；

4. 幻觉抑制 Prompt 固定模板：强制 AI 仅使用提供文档作答，无匹配资料统一回复 “暂无对应文档，无法解答”，输出末尾附上文档名称 \+ 页码来源。

### 4\.3 自定义插件开发（企业业务定制功能）

1. 工单分析插件：对接业务 MySQL，自动抓取历史同类故障工单，补充检索上下文；

2. 备件查询插件：检索到零部件后自动调取采购库存数据；

3. 导出插件：将 AI 生成的维修方案导出 Word，同步至飞书文档；

### 4\.4 应用搭建

1. 内部员工通用问答 Bot（网页端嵌入企业 OA）；

2. 飞书机器人：员工在部门群直接提问知识库；

3. 售后工单自动处理工作流：工单提交→自动检索故障方案→回填工单系统。

### 交付物

自定义 Python 插件源码、Dify 工作流配置备份、对外 OpenAPI 接口文档、SSO 对接改造包。

## 阶段 5：全链路测试、安全合规验收（5\~8 天，测试 \+ 合规部门参与）

### 5\.1 功能测试

1. 权限测试：售后账号无法检索研发涉密文档；

2. 边界测试：模糊故障描述、不存在的设备型号，验证 AI 不编造；

3. 多轮对话测试：上下文记忆不溢出，长对话准确引用前文设备信息；

4. 并发压测：200 用户同时问答，响应稳定无超时。

### 5\.2 合规 \& 安全测试（企业硬性验收项）

1. 审计日志测试：所有问答完整记录，支持按人员、时间筛选导出；

2. 数据安全：向量库、文档库加密存储，无明文泄露；

3. 内容安全：内置敏感词过滤，屏蔽设备涉密工艺外泄；

4. 离线验证：断开外网，系统全部功能正常运行。

### 5\.3 自动化评测流水线搭建

基于 Dify 评测功能，导入 500 条业务标准问题，每周自动运行，输出检索准确率、幻觉率报表，持续优化分块、检索参数。

### 交付物

《系统测试报告》《合规验收报告》《自动化评测脚本》

## 阶段 6：上线、运维、迭代（长期持续）

### 6\.1 灰度上线

1. 先交付售后部试点使用 2 周，收集维修工程师反馈；

2. 修复检索漏检、回答冗长等问题后，全公司开放。

### 6\.2 运维监控体系（LLMOps 能力）

Dify 内置监控 \+ 自研监控脚本：

1. 业务指标：问答量、检索平均时延、拒绝回答占比（幻觉控制指标）；

2. 资源指标：Milvus 内存占用、大模型 GPU 负载、磁盘存储增量；

3. 告警机制：接口超时、向量库异常、大批量文档同步失败推送运维消息。

### 6\.3 持续迭代优化

1. 每周增量同步新增业务文档；

2. 根据用户提问日志，优化分块规则、补充专业术语词典；

3. 每季度更新 Embedding、重排模型版本，提升检索精度；

4. 业务新增需求扩展工作流、插件（如新增质检 AI 助手）。

### 交付物

《运维手册》《系统监控看板》《迭代优化记录》

# 配套补充：该项目真实区分 Dify/Coze 使用边界

1. 本项目为什么不用 Coze（扣子） 集团内网离线、强合规、涉密技术资料、需要深度私有化二次开发；Coze SaaS 依赖火山方舟公网，开源版权限、审计、向量调优能力薄弱，无法满足制造企业安监内网要求；

2. Coze 仅适用本集团边缘轻量场景 对外客服抖音机器人、公开产品咨询，无涉密内部资料，使用 Coze 快速搭建对外 Bot，和内网 Dify 知识库物理隔离；

3. 框架分工 平台层：Dify 承载全部核心 RAG 业务；底层研发：少量定制化逻辑使用 LlamaIndex 开发同步脚本、文档处理工具。
