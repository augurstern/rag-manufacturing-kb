#### 4.1.3 Docker Compose 编排

**文件**：`docker-compose.yml`（项目根目录）

```yaml
version: '3.8'

services:
  backend:
    build: ./backend
    container_name: rag-backend
    ports:
      - "8000:8000"
    volumes:
      - ./chroma_db:/app/chroma_db      # ChromaDB 持久化
      - ./data:/app/data                 # 测试文档
      - ./logs:/app/logs                 # 日志文件
    environment:
      - DEEPSEEK_API_KEY=${DEEPSEEK_API_KEY}
      - DEEPSEEK_BASE_URL=${DEEPSEEK_BASE_URL:-https://api.deepseek.com}
      - SQLITE_DB_PATH=/app/data/rag_system.db
      - CHROMA_PERSIST_DIR=/app/chroma_db
      - LOG_LEVEL=${LOG_LEVEL:-INFO}
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

  frontend:
    build: ./frontend
    container_name: rag-frontend
    ports:
      - "8501:8501"
    environment:
      - BACKEND_URL=http://backend:8000    # 容器内通过服务名通信
    depends_on:
      backend:
        condition: service_healthy
    restart: unless-stopped

  # 注意：ChromaDB 使用 PersistentClient 模式，直接在 backend 进程内运行
  # 不需要单独的 chromadb 服务容器
  # 如果未来需要独立扩展 ChromaDB，可以添加独立服务
```

**面试时关于"为什么 ChromaDB 没有独立容器"的解释**：
> ChromaDB 的 PersistentClient 模式在单机部署时足够，向量数据通过 Volume 挂载持久化。
> 生产环境如果用 Milvus，则需要独立的 Milvus 集群（Coordinator + Proxy + Data Node + Index Node），
> 那才需要单独的服务编排。

### 4.2 云服务器部署（2 天，自习日）

**操作流程**：

```bash
# === 在本地 ===
# 1. 将代码推送到 GitHub/Gitee
git remote add origin <你的仓库地址>
git push -u origin main

# === 在云服务器上 ===
# 2. SSH 登录
ssh user@your-server-ip

# 3. 安装 Docker Compose（如果没有）
sudo apt update
sudo apt install docker-compose -y

# 4. 克隆项目
cd ~
git clone <你的仓库地址> rag-project
cd rag-project

# 5. 配置环境变量
cp .env.example .env
nano .env   # 填入 DEEPSEEK_API_KEY

# 6. 启动
docker-compose up -d

# 7. 检查状态
docker-compose ps
docker-compose logs backend | head -20

# 8. 配置防火墙（开放 8501 端口）
# 云服务商控制台 → 安全组 → 添加入站规则：TCP 8501
# Ubuntu 自带 ufw 可能也需要：
sudo ufw allow 8501

# 9. 验证
curl http://localhost:8000/health
# 浏览器访问：http://<服务器公网IP>:8501
```

**部署检查清单**：
- [ ] SSH 到服务器，`docker-compose ps` 显示两个服务都是 `Up`
- [ ] 公网 IP:8501 可以打开 Streamlit 界面
- [ ] 上传一个测试文档，提问正常返回
- [ ] 重启服务器后 `docker-compose up -d` 数据仍在（Volume 持久化）

**常见部署问题**：

| 问题 | 原因 | 解决 |
|------|------|------|
| `docker-compose` 命令不存在 | Docker Compose 未安装或版本是 `docker compose`（无横杠） | 用 `docker compose` 试试；或用 `apt install docker-compose-plugin` |
| 容器启动后立即退出 | 环境变量缺失或代码报错 | `docker-compose logs backend` 查看错误日志 |
| 公网 IP 无法访问 | 防火墙/安全组未开放端口 | 检查云服务商控制台安全组规则 |
| 内存不足 | 4G 内存跑三个服务可能吃紧 | 关闭不必要的系统服务；chromadb 用轻量模式 |

```python
# 文件类型白名单
ALLOWED_EXTENSIONS = {".pdf", ".docx", ".txt"}
ALLOWED_MIME_TYPES = {
    "application/pdf",
    "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
    "text/plain",
}

# 文件大小限制
MAX_FILE_SIZE = 20 * 1024 * 1024  # 20MB

# 文件名清洗（防止路径穿越攻击）
import os
import uuid

def secure_filename(filename: str) -> str:
    """生成安全的文件名"""
    ext = os.path.splitext(filename)[1].lower()
    return f"{uuid.uuid4().hex}{ext}"
```

#### 4.3.3 请求频率限制（简单版）

```python
# 基于 IP 的简单限流器（生产环境用 Redis + 滑动窗口）
from collections import defaultdict
import time

class SimpleRateLimiter:
    def __init__(self, max_requests: int = 30, window_seconds: int = 60):
        self.max_requests = max_requests
        self.window = window_seconds
        self.requests: dict[str, list[float]] = defaultdict(list)

    def is_allowed(self, client_ip: str) -> bool:
        now = time.time()
        # 清理过期记录
        self.requests[client_ip] = [
            t for t in self.requests[client_ip]
            if now - t < self.window
        ]
        if len(self.requests[client_ip]) >= self.max_requests:
            return False
        self.requests[client_ip].append(now)
        return True
```

### 4.4 日志系统（1.5 天）

**文件**：`backend/app/services/logger.py`

```python
import logging
import sys
from pathlib import Path

def setup_logger(log_level: str = "INFO", log_dir: str = "./logs"):
    """配置应用日志"""
    Path(log_dir).mkdir(parents=True, exist_ok=True)

    logger = logging.getLogger("rag_system")
    logger.setLevel(getattr(logging, log_level.upper()))

    # 格式
    formatter = logging.Formatter(
        "%(asctime)s [%(levelname)s] %(name)s - %(message)s",
        datefmt="%Y-%m-%d %H:%M:%S",
    )

    # 控制台输出
    console_handler = logging.StreamHandler(sys.stdout)
    console_handler.setFormatter(formatter)
    logger.addHandler(console_handler)

    # 文件输出（按天轮转）
    file_handler = logging.FileHandler(f"{log_dir}/rag_{datetime.now():%Y%m%d}.log")
    file_handler.setFormatter(formatter)
    logger.addHandler(file_handler)

    return logger
```

**关键日志点**：
- 上传成功/失败时记录：文件名、大小、分块数、耗时
- 问答请求时记录：问题、检索结果数、LLM 生成耗时、总耗时
- API 异常时记录：完整错误堆栈

### 4.5 错误处理与健壮性（1.5 天）

| 场景 | 处理方式 |
|------|----------|
| LLM API 调用失败 | 重试 2 次（指数退避 1s → 2s）；仍失败返回友好错误信息"AI 服务暂时不可用，请稍后重试" |
| ChromaDB 连接失败 | 启动时检测，不可用则报错退出并提示检查路径 |
| 文档解析失败 | 返回具体原因（"PDF 文件损坏"、"不支持该 Word 版本"）而非笼统的"解析失败" |
| 前端连不上后端 | Streamlit 页面显示"后端服务不可用，请确认已启动后端服务"而非白屏 |
| 磁盘空间不足 | 上传时检查可用空间，不足时拒绝上传并提示清理 |
