# 面试补课清单 —— 简历写的 vs 你实际做的

**目的**：你实际写的代码是简化版，但简历上写的是完整版。这份清单列出了每个项目你需要**系统学习、记笔记、能口述**但不需要写代码的知识点。

## 使用说明

## 项目一：RAG 知识库 补课清单

### 1. LangChain / LlamaIndex ⭐⭐⭐

**为什么需要补**：简历写了 LangChain/LlamaIndex，但你没实际用。面试官大概率问"你用 LangChain 做了什么"。

**要学到什么程度**：
- 能说出 LangChain 的核心模块：Document Loader、Text Splitter、VectorStore、RetrievalQA、Chain
- 能用 2 分钟讲清楚"为什么我选了手写而不是 LangChain"（你已经有了——面试时说：手写能精确控制每个环节，LangChain 黑盒调试困难）
- 知道 LangChain 的 LCEL（LangChain Expression Language）是什么（一句话：用 `|` 管道符串联多个组件）
- 知道 LlamaIndex 和 LangChain 的区别（LlamaIndex 偏数据索引和检索，LangChain 偏通用 LLM 应用编排）

**学习资源**：
- LangChain 官方文档首页 + Quick Start（2 小时）
- LlamaIndex 官方文档首页（30 分钟了解定位）

**自行验证**：找同学问"你用 LangChain 做了什么"，不看笔记回答。

### 2. Milvus 向量数据库 ⭐⭐⭐

**为什么需要补**：简历写了 Milvus，但你实际用 ChromaDB。面试官大概率问"Milvus 和 ChromaDB 的区别"。

**要学到什么程度**：
- Milvus 的核心架构：Coordinator（协调）+ Proxy（代理）+ Data Node（数据）+ Index Node（索引），知道为什么分这么多角色（分布式、高可用、性能隔离）
- 索引类型：IVF_FLAT、HNSW、IVF_PQ 的区别（速度 vs 精度的 trade-off）
- 能对比 ChromaDB 和 Milvus：ChromaDB 单机能用但无分布式能力、无原生多租户、无索引类型选择
- 知道 Milvus 最低部署要求（8GB 内存），以及为什么你的 Demo 选了 ChromaDB

**学习资源**：
- Milvus 官方文档"Architecture Overview"（1 小时）
- Milvus 和 ChromaDB 的对比博客（搜 2-3 篇中文对比文）

### 3. Self-RAG 自检索纠错 ⭐⭐

**为什么需要补**：简历写了 Self-RAG，面试可能问"你的 Self-RAG 怎么实现的"。

**要学到什么程度**：
- Self-RAG 的核心思想：让 LLM 在生成过程中自己判断"是否需要检索更多信息"
- 和普通 RAG 的区别：普通 RAG 是"先检索再生成"，Self-RAG 是"生成过程中随时可能再次检索"
- 实现的关键：在 Prompt 中加入反思标记（`<retrieve>`、`<critique>`），让 LLM 自己判断是否信息不够
- 坦诚说你的实际的层次：做了来源引用和幻觉拒绝，但没有做完整的 Self-RAG 自检索循环

**学习资源**：
- Self-RAG 论文的中文解读博客（搜 1-2 篇，各花 30 分钟）

## 项目二：Agent 工具调用 补课清单

### 6. LangChain AgentExecutor ⭐⭐⭐

**为什么需要补**：简历写了 LangChain Agent，面试大概率问"你用 LangChain 怎么构建 Agent 的"。

**要学到什么程度**：

  from langchain.agents import initialize_agent, AgentType

      llm=llm,

- 知道 `AgentType` 的几种常见类型（ZERO_SHOT_REACT、CHAT_CONVERSATIONAL_REACT、OPENAI_FUNCTIONS）
- 能解释你为什么手写而不是用 LangChain：手写能控制解析逻辑——LangChain 的默认 OutputParser 对中文和自定义工具支持不够好

### 7. Function Calling 机制 ⭐⭐⭐

**为什么需要补**：这是 Agent 的核心底层机制，面试必问。你的 Agent 用的是 ReAct 范式，不是原生 Function Calling，所以需要单独学。

**要学到什么程度**：
- Function Calling 的原理：LLM 不再输出自然语言，而是输出一个 JSON 对象 `{name: "web_search", arguments: {query: "..."}}`
- 和 ReAct 的区别：Function Calling 是 LLM 原生支持的（模型专门训练过），ReAct 是靠 Prompt 引导（任何 LLM 都能用）
- 优缺点：Function Calling 格式稳定（模型经过了指令微调），但只支持特定模型；ReAct 通用但解析容易出错
- DeepSeek API 支持 Function Calling 吗？当前（2026 年初）不完全支持，所以选用 ReAct 方式

**学习资源**：
- OpenAI 官方 Function Calling 文档（1 小时）
- 一篇"Function Calling vs ReAct"对比博客

### 8. Docker 沙箱执行 ⭐⭐

**为什么需要补**：简历写了"高危代码放入 Docker 沙箱执行"。

**要学到什么程度**：

### 9. 双记忆架构 ⭐⭐

**为什么需要补**：简历写了"短期记忆（Redis）+ 长期记忆（向量库）"。

**要学到什么程度**：
- 短期记忆：当前对话的前 N 轮消息，存在 Redis 中，设 TTL 过期
- 长期记忆：把历史对话中的重要信息向量化存入向量库，下次对话时先检索相关历史
- 为什么要分开存：短期记忆追求速度（Redis 微秒级），长期记忆追求语义检索（向量库毫秒级但能跨会话检索）
- 坦诚：你的 Demo 只做了单会话记忆（messages 列表），这是基础版。双记忆架构的原理你理解，但需要更多工程投入

## 项目三：多 Agent 协作 补课清单（补课量最大）

### 10. LangGraph 状态图 ⭐⭐⭐

**为什么需要补**：简历写了 LangGraph，这是多 Agent 最核心的技术关键词。面试大概率问"你用 LangGraph 做了什么"。

**要学到什么程度**：
- LangGraph 的核心概念（能用白板画出来）：
  - **StateGraph**：定义整个应用的状态（一个 dict）
  - **Node（节点）**：处理逻辑的最小单元（每个 Worker Agent 是一个节点）
  - **Edge（边）**：节点之间的连接。普通边（固定流向）和条件边（根据状态决定流向）
  - **Checkpoint**：每个节点执行后自动保存状态，支持暂停→恢复→回滚
- LangGraph 的标准代码结构（不需要会写，但要能看懂）：
  ```python
  from langgraph.graph import StateGraph, END
  
  workflow = StateGraph(AgentState)  # 定义状态
  workflow.add_node("researcher", researcher_node)  # 加节点
  workflow.add_node("analyst", analyst_node)
  workflow.add_conditional_edges("researcher", decide_next, {  # 条件边
      "continue": "analyst",
      "end": END,
  })
  workflow.set_entry_point("researcher")  # 入口
  app = workflow.compile()  # 编译成可执行的应用
  ```
- LangGraph 和 LangChain 的关系：LangGraph 是 LangChain 的子项目，专门处理有状态的、多步骤的 Agent 工作流
- 你是怎么做的：手写了 Supervisor 调度逻辑（拆分→执行→汇总），和 LangGraph 在概念上是通的但没有用它的状态图引擎

**学习资源**：
- LangGraph 官方 Quick Start + 前 3 个 Tutorial（3-4 小时）
- 一篇 "LangGraph 入门教程" 中文博客（1 小时）

**自行验证**：对着白板画 LangGraph 的状态流转图，标出节点、边、条件判断。

### 12. LLaMA 私有化部署（Ollama/vLLM）⭐⭐⭐

**为什么需要补**：简历写了本地 LLaMA 部署，这是政企项目的硬需求。这也是**最容易穿帮的知识点**——因为硬件原因你完全没跑过。

**要学到什么程度**：

**Ollama**（至少亲手部署一次）：

### 13. Agent 间通信协议（A2A）⭐⭐

**为什么需要补**：简历写了"自定义标准化 Agent 通信协议"。

**要学到什么程度**：
- 多 Agent 通信的三种方式：
  1. **自然语言**：Agent A 用自然语言告诉 Agent B 要做什么（灵活但不精确）
  2. **JSON Schema**：定义固定的 JSON 格式，如 `{from, to, task_type, payload, priority}`
  3. **结构化对象**（推荐）：Pydantic 定义消息结构，`class AgentMessage(BaseModel): from_agent, to_agent, task, data`
- 为什么需要固定协议：避免自由文本解析错误、支持类型检查、方便日志记录
- 你的做法：简化版用的是 Python dict 传递消息，概念上和 JSON Schema 一致

### 14. 多 Agent 故障处理 ⭐⭐

**为什么需要补**：简历写了"任务中断续跑、失败回滚"。这是体现"工作经验"的关键细节。

**要学到什么程度**：
- 多 Agent 系统常见的 4 种故障：
  1. **任务丢失**：Worker 超时或崩溃，子任务没被执行 → 解决：任务队列 + 超时重分配
  2. **死循环**：Supervisor 反复分配同一任务 → 解决：最大重试次数 + 重试指数退避
  3. **结果冲突**：两个 Worker 给出矛盾的信息 → 解决：投票机制 或 人工介入判断
  4. **格式异常**：Worker 输出不符合 JSON Schema → 解决：格式校验 + 拒绝后重新请求
- Checkpoint（断点续跑）的实现思路：每个节点执行完后自动序列化当前状态（LangGraph 自带这个功能）
- 你的做法：简化版只在单次调用内做了 try/except。但理解了上述生产级方案

**学习资源**：
- 一篇"Multi-Agent 系统的常见问题与解决"博客

### 15. RabbitMQ 消息队列 ⭐

**为什么需要补**：简历写了 RabbitMQ 异步分发。

**要学到什么程度**：
- RabbitMQ 在多 Agent 系统中的角色：任务队列（Supervisor 发任务到队列，Worker 从队列取任务）
- 为什么需要消息队列：解耦、削峰、异步、失败重试
- 基础概念：Producer（发消息）、Consumer（收消息）、Queue（队列）、Exchange（交换机）
- 坦诚：Demo 中 Worker 是同步顺序执行的（Python 函数调用），生产环境用 RabbitMQ 做异步解耦

## 面试中"坦诚"的话术模板

当被问到简历上有但你实际没用过的技术时：

> "我的 Demo 中用了 [实际用的]，选它是为了 [轻量/快速验证/学习底层原理]。
> 但我深入研究了 [简历写的那个]，知道它的核心优势是 [A/B/C]。
> 如果从 Demo 迁移到生产，我会换 [简历写的那个]，因为 [D/E/F] 场景需要它。"

这个模板同时展示了：你的实际能力 + 你的技术视野 + 你的设计决策能力。比硬说自己用过但答不上来，要好得多。
