### 5.2 面试话术准备（3 天）

#### 5.2.1 项目介绍三板斧（1 天）

**30 秒电梯演讲版**（面试官说"简单介绍一下你的项目"）：

> "我做过一个制造业内部知识库的 RAG 系统。用 FastAPI 做后端，Streamlit 做前端，
> ChromaDB 做向量存储，DeepSeek 提供大模型能力。核心实现了文档解析、制造业专用分块策略、
> 混合检索（向量+关键词）、以及带来源引用的问答生成。整套系统 Docker Compose 部署，
> 在云服务器上可以直接跑。这个项目让我从零理解了 RAG 的全链路原理。"

**2 分钟详细版**（面试官说"展开讲讲"）：

> （结构：背景 → 架构 → 核心亮点 → 技术选型理由）
>
> "这个项目的背景是帮一个制造集团做内部知识库。他们有几万份分散的文档——维修手册、
> 操作规范、零部件参数——员工查资料平均要 40 分钟。传统关键词搜索搜不到专业术语，
> 直接用大模型又会瞎编参数，所以他们需要一个 RAG 系统。
>
> 我做的系统分三层——文档处理层、检索层、生成层。文档处理支持 PDF/Word/扫描件，
> 解析后按制造业的特点做分块：普通文档走 500 字滑动窗口，维修手册用父子分块
> （父块 3000 字检索保证语义完整，子块 500 字送入 LLM），表格按行分块防止参数混淆。
> 检索层做了混合检索——向量相似度 + BM25 关键词，用 RRF 算法融合两路结果。
> 生成层设计了一套 Prompt 模板，强制 AI 只用文档内容回答、附上来源页码、
> 没有答案就直接拒绝不编造。
>
> 技术栈上，向量库用的是 ChromaDB，选它是因为单机部署简单、学习成本低。
> 如果是生产环境会换 Milvus 做分布式。大模型用了 DeepSeek API，生产环境会换成本地部署的私有模型，
> 因为企业数据不能出内网。整套系统通过 Docker Compose 部署，在 4 核 4G 的云服务器上就能跑。"

**5 分钟深度版**（面试官说"聊聊你的技术决策"）：

> （在 2 分钟版基础上，展开以下话题，每个 2-3 分钟）：
> - 分块策略的 trade-off（chunk 太大检索不准，太小语义碎片化）
> - 混合检索的 RRF 算法细节
> - Prompt 的设计迭代过程
> - 从 CLI 到 Web 到 Docker 的开发流程
> - 如果重新做一遍会怎么改进

#### 5.2.2 高频面试问题准备（2 天）

**准备方式**：每道题写 200-300 字的回答要点（不要背全文，背要点 + 临场组织语言）。

| 类别 | 问题 | 回答要点 |
|------|------|----------|
| RAG 原理 | "什么是 RAG？为什么要用？" | 大模型知识有截止日期 + 幻觉问题 + 企业数据不出内网；RAG=检索+生成，检索找事实，生成组织语言 |
| RAG 流程 | "RAG 的完整流程是怎样的？" | 文档解析→分块→向量化→存入向量库→用户提问向量化→相似度检索→取 Top K→拼 Prompt→LLM 生成 |
| 分块策略 | "你怎么确定分块大小？" | 500 字是经验值 + 制造业文档特点 + 重叠防止边界截断 + 如果重新做可以用 RAGAS 评测不同 chunk size |
| 父子分块 | "父子分块和普通分块有什么区别？" | 父块保证语义完整、子块保证精确；维修手册场景：一个故障描述跨多段必须用父块检索 |
| 向量检索 | "向量检索的原理是什么？" | Embedding 将文本映射到高维向量空间 → 用余弦相似度衡量语义距离 → HNSW 索引加速搜索 |
| BM25 | "BM25 是怎么工作的？" | 基于词频+逆文档频率的经典算法；IDF 降低"的""了"等高频词权重；BM25 对专业术语精确匹配尤其有效 |
| 混合检索 | "为什么混合检索比单一检索好？" | 互补：向量=语义近似，BM25=精确匹配；制造业术语"轴承座"vs"轴承盖"语义近似但不可互换 |
| RRF | "RRF 融合算法怎么算的？" | RRF(d)=∑1/(k+rank_i)；k=60 平滑；只用排名不用分数，天然消除不同检索器分数量纲差异 |
| Embedding | "为什么选 DeepSeek Embedding？" | 4096 维效果最好 + 成本低+兼容 OpenAI SDK 调用；对比方案：BGE 系列（本地免费但维度低） |
| 向量库 | "ChromaDB 和 Milvus/FAISS 的区别？" | FAISS 是纯向量索引库(没有元数据)，ChromaDB 自带元数据+简单好用，Milvus 分布式+高并发+企业级。本项目用 ChromaDB 适合单机 Demo，生产迁移 Milvus |
| Prompt | "你的 Prompt 怎么设计的？" | 四规则：只用文档内容、附来源引用、禁止编造、专业简洁。每点有明确设计意图 |
| 幻觉 | "怎么防止 RAG 产生幻觉？" | 多层机制：Prompt 约束 + 低 temperature + 无匹配拒绝回答 + 简单术语校验 |
| 评估 | "你怎么评估 RAG 系统好不好？" | 手动测试集：检索准确率（Top K 是否相关）、回答质量（是否准确+有无幻觉+来源是否正确）。生产环境会用 RAGAS 自动化评测 |
| 部署 | "Docker Compose 部署的架构？" | backend(8000)+frontend(8501)+ChromaDB(进程内)；Volume 持久化；depends_on 控制启动顺序；健康检查 |
| 性能 | "问答延迟大概多少？瓶颈在哪？" | 总延迟≈检索500ms + LLM生成2-4s；瓶颈在 LLM；优化方向：流式输出（体感首 token<2s）、减少 context 长度 |
| 业务 | "制造业知识库和通用知识库有什么不同？" | 文档类型特殊（扫描件多+表格多）、术语专业、对准确性要求极高（编造参数=安全事故）、需要来源追溯 |
| 踩坑 | "项目中遇到最大的困难是什么？" | (准备 2-3 个自己的真实踩坑经历，每个 2-3 分钟) |
| 改进 | "如果重新做，你会怎么改进？" | 加入自动化评测(RAGAS)、增加多轮对话、改用 Milvus+PostgreSQL、加 BGE-Reranker、做 A/B 测试 |

**练习方式**：
1. 每道题先写要点
2. 对着录音讲一遍（3-5 分钟/题）
3. 回听：语气词多不多？逻辑清不清晰？有没有跑题？
4. 重录直到流畅

**交付物**：
- [ ] 18 道高频题的核心要点笔记
- [ ] 每道题至少口头练习 2 遍

### 5.7 "知道但没用到"清单（0.5 天）

**目的**：面试被问到"你为什么不用 XXX"时的回答素材。

| 技术 | 为什么没用 | 面试话术 |
|------|-----------|----------|
| LangChain | 封装太厚，学习阶段手写能更深入理解 | "我用 FastAPI 手写整个 Pipeline，这样对每个环节的理解更深。LangChain 的抽象在复杂场景下反而难调试。" |
| LlamaIndex | 同上 | "LlamaIndex 适合快速原型，但我想深入理解 RAG 每个环节，所以选择手写。" |
| Milvus | 单机部署太重 | "ChromaDB 单机足够，而且演示环境 4G 内存跑不动 Milvus。但我知道生产环境需要 Milvus 的分布式能力和索引性能。" |
| RAGAS | 项目规模不大，优先手动评测 | "我用 30 个手动测试问题做评估。RAGAS 适合大规模自动化评测，后续迭代会引入。" |
| Dify | 平台化太重，不便于展示编码能力 | "Dify 是低代码平台，适合快速搭建但对原理理解帮助不大。我选择手写全链路来证明我的编码和设计能力。" |
| Redis | 单机场景不需要独立缓存 | 用内存字典做了简单限流，生产环境会引入 Redis 做缓存+限流+会话管理 |
